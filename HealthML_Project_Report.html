All three services run in isolated Docker containers orchestrated by Docker Compose and deployed on a GCP Compute Engine instance. Nginx serves the compiled React SPA and reverse-proxies /api/* to the Express backend, which in turn proxies ML-heavy operations (train, explain, fairness, certificate) to the Python FastAPI service. State is managed client-side via a single React Context; the ML service keeps trained models in memory keyed by UUID.

3 · Key Design Decisions
Decision 1
Real scikit-learn, not simulated
All model training uses actual scikit-learn pipelines. This means metrics are genuine and vary with data and hyperparameters — demonstrating real ML trade-offs to learners rather than hard-coded numbers.
Decision 2
Microservice split (3 containers)
Separating Python ML from the Node.js API allows independent scaling and avoids bundling a Python runtime into the Node image. The backend acts as a thin proxy, keeping each service's responsibility clear.
Decision 3
Ablation-based explainability
Per-patient explanations replace each feature with its training-set mean and measure the change in predicted probability. This works across all 6 model types without requiring SHAP or LIME dependencies.
Decision 4
SMOTE on training set only
Synthetic rows are tagged with __smote__ and excluded from the test split, preventing data leakage. This was a critical correctness fix discovered during Sprint 4 QA review.
Decision 5
10 pp fairness threshold
A bias alert is triggered when any subgroup's sensitivity deviates from the overall sensitivity by more than 10 percentage points — calibrated to avoid noise from small datasets while catching clinically meaningful gaps.
Decision 6
Step access control
Users cannot skip steps without completing prerequisites (schema validation before Step 3; data preparation before Step 4). This mirrors real ML workflow constraints and enforces a correct learning sequence.

4 · Challenges & Solutions
1
SMOTE Data Leakage
Initially SMOTE was applied before the train/test split, inflating test-set metrics. Fixed by tagging synthetic rows with a sentinel column and filtering them out of the test split during training.
2
20-Domain Coverage Consistency
Each domain required domain-specific clinical sense-check text, feature label mappings, and fairness thresholds. Managed via explicit mapping tables in the ML service and frontend context.
3
PDF Certificate Unicode Errors
The built-in Helvetica font in fpdf2 does not support Unicode. Resolved by a sanitisation helper (_s()) that replaces em-dashes, arrows, and special characters before rendering.
4
Normalised Age Fairness Bands
After z-score normalisation, raw age bands (18–60, 61–75, 76+) produced empty groups. Fixed by detecting normalised values (max < 10) and switching to quantile-based tertile splits.
5
Cross-Container Networking (Docker)
The backend used localhost for the ML service, which fails inside Docker. Resolved by setting ML_HOST=ml via an environment variable in docker-compose, using Docker's internal DNS.
6
SVM / KNN Feature Importance
Tree-based models expose feature_importances_; SVM, KNN, and Naive Bayes do not. Added a permutation importance fallback using the test set, keeping the Explainability step consistent across all 6 models.
7
Training Latency Under 3 Seconds
Random Forest with 300 trees on large datasets exceeded the 3 s target. Capped trees at 300 maximum and added a latency display in the UI so users observe the speed vs. accuracy trade-off directly.
8
Sprint 4 Bias Threshold Regression
Two conflicting thresholds (8 pp and 15 pp) existed simultaneously. Unified to a single >10 pp rule with explicit edge-case tests at exactly 10 pp and 11 pp in the Sprint 4 acceptance suite.
5 · Lessons Learned
✓
Test early, test real data
Several bugs (SMOTE leakage, Unicode PDF errors) were invisible with synthetic data but appeared immediately with real CSV uploads. User testing in Sprint 5 with real health datasets was invaluable.
✓
Gate reviews prevent debt
Enforcing Sprint Gate criteria (live demo, passing acceptance tests) before marking a sprint complete prevented unfinished work from accumulating across sprints.
✓
Microservices add complexity
While the 3-service architecture is clean, debugging cross-container issues (networking, environment variables) cost significant time. For a team of this size, a monorepo approach would have been simpler in earlier sprints.
✓
Plain language matters
User testing revealed that clinicians ignored metric values until labels were rephrased (e.g. "% of truly positive patients caught"). The ML glossary and plain-language UI copy were added as a direct result.
✓
Fairness is not an afterthought
Adding subgroup analysis in Sprint 4 required retrofitting bias data into training outputs already consumed by Step 5. Building fairness hooks from Sprint 2 would have been significantly cheaper.
✓
Agile velocity improved
Sprint 1 delivered 3 story points; Sprint 5 delivered 11. Weekly showcases with instructor feedback accelerated decision-making and reduced rework in later sprints.
